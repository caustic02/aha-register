# State of the Art: Internationalization (i18n)

> Last updated: 2026-03-15
> Status: ACTIVE

## What This Is
Full bilingual EN/DE support via i18next. Every user-facing string goes through the `useAppTranslation` hook. Both locale files must stay in parity.

## Architecture

### Library Stack
- `i18next` ‚Äî core translation engine
- `react-i18next` ‚Äî React integration via `initReactI18next` plugin
- `expo-localization` ‚Äî device locale detection
- `compatibilityJSON: 'v4'` ‚Äî i18next v4 JSON format

### Initialisation (`src/i18n/index.ts`)
```
i18n.use(initReactI18next).init({
  lng: detectLanguage(),
  fallbackLng: 'en',
  supportedLngs: ['en', 'de'],
  resources: { en: { translation: en }, de: { translation: de } },
  interpolation: { escapeValue: false },
  compatibilityJSON: 'v4',
})
```

### Language Detection
`detectLanguage()` runs at init time (not lazily):
1. Calls `expo-localization` `getLocales()` ‚Üí takes `languageCode` from the first locale entry
2. Strips region suffix: `'en-US'` ‚Üí `'en'`, `'de-AT'` ‚Üí `'de'`
3. Checks against `SUPPORTED_LANGUAGES = ['en', 'de']`
4. Falls back to `'en'` if device language is unsupported

No runtime language switching ‚Äî language is set once at app start. Changing language requires app restart (via Settings screen which updates the i18next `lng`).

### Bundle Strategy
Both locale JSON files are imported statically at build time. There is no lazy loading or remote translation fetching. This means:
- Zero network dependency for translations
- Both locales shipped in every bundle
- Adding a new locale requires a code change and rebuild

### Interpolation
Variables use `{{variableName}}` syntax. Examples from the locale files:
- `"header_count": "{{count}} objects"`
- `"last_synced": "Last synced: {{time}}"`
- `"delete_confirm": "Delete \"{{title}}\"? ..."`

`escapeValue: false` ‚Äî React's JSX rendering handles XSS escaping, so i18next does not double-escape.

### Hook
All components use `useAppTranslation` (`src/hooks/useAppTranslation.ts`) ‚Äî a thin wrapper around `react-i18next` `useTranslation`. Never call `useTranslation` directly.

## Key Files

| File | Purpose |
|------|---------|
| `src/i18n/index.ts` | i18next configuration and language detection |
| `src/i18n/locales/en.json` | English translations (464 keys) |
| `src/i18n/locales/de.json` | German translations (464 keys) |
| `src/hooks/useAppTranslation.ts` | Shared hook ‚Äî use this, not useTranslation directly |

## Rules

- Every user-facing string must use `useAppTranslation`
- EN + DE keys added in the same commit ‚Äî never merge an EN-only addition
- German: natural conversational tone, not formal/stiff (`du` not `Sie`)
- Key naming: `namespace.key` (e.g. `capture.sha256`, `objects.delete_title`)
- Run parity check before every commit: both files must have identical key count and key set

## Known Gaps

- No RTL language support
- No pluralization rules configured (i18next `count`-based plurals not used ‚Äî strings hand-written)
- Language change at runtime requires full i18next `changeLanguage()` call; currently only triggered by Settings screen
- No translation memory or professional review ‚Äî German written by bilingual developer
- Third language (FR, ES, etc.) would require adding to `SUPPORTED_LANGUAGES`, new locale file, and detection logic update
