# Database

> Last updated: 2026-03-15
> Status: Active

## What This Is

SQLite database layer for aha! Register, now with SQLCipher AES-256
full-database encryption. All records, media references, audit trails,
and sync queue entries are encrypted at rest on device.

## Architecture

```
AppShell
  └─ initDatabase()                      ← src/db/database.ts
       ├─ getOrCreateDatabaseKey()       ← src/utils/db-encryption.ts
       │    └─ SecureStorage (Keychain / Keystore)
       ├─ [first launch] migrateToEncryptedDatabase()
       │    └─ export plaintext → delete → create encrypted → reimport
       ├─ PRAGMA key = "x'<256-bit hex>'"  ← MUST be first operation
       ├─ PRAGMA journal_mode=WAL
       ├─ PRAGMA foreign_keys=ON
       ├─ SCHEMA_SQL (14 tables, IF NOT EXISTS)
       └─ INDEXES_SQL (19 indexes)
```

### Encryption Key Lifecycle

```
First launch:
  expo-crypto → 32 random bytes → hex encode → SecureStorage.setItem()

Every launch:
  SecureStorage.getItem() → PRAGMA key → database unlocked
```

The key is:
- 256 bits of cryptographic randomness (not derived from a passphrase)
- Stored in iOS Keychain / Android Keystore (hardware-backed)
- Never logged, never included in crash reports, never transmitted
- Unrecoverable if SecureStore is wiped (inherent trade-off of device encryption)

### Migration (plaintext → encrypted)

On the first launch after enabling SQLCipher, if the database already
contains data, the migration runs automatically:

1. Open existing DB without PRAGMA key (SQLCipher reads plaintext)
2. Export all rows from all 14 tables into memory
3. Close and delete the plaintext file
4. Create a new encrypted DB with the same name
5. Apply schema + indexes, reimport all data
6. Verify object count matches
7. Set `_db_encrypted` flag in SecureStorage

The migration flag ensures this runs exactly once.

## Key Files

| File | Role |
|------|------|
| `src/db/database.ts` | `initDatabase()` — opens DB, sets PRAGMA key, runs schema |
| `src/db/schema.ts` | 14-table schema definition (SACRED — do not modify) |
| `src/db/indexes.ts` | 19 index definitions |
| `src/utils/db-encryption.ts` | `getOrCreateDatabaseKey()` — generates/retrieves 256-bit key |
| `src/utils/db-migration-encrypt.ts` | One-time plaintext → encrypted migration |
| `src/utils/secure-storage.ts` | SecureStorage adapter (iOS Keychain / Android Keystore) |
| `app.json` | `["expo-sqlite", { "useSQLCipher": true }]` plugin config |
| `android/gradle.properties` | `expo.sqlite.useSQLCipher=true` (generated by prebuild) |

## Decision History

- **2026-03-15** — Enabled SQLCipher via `expo-sqlite` plugin. Chose raw hex
  key (no PBKDF2 derivation) for faster DB opens and full 256-bit entropy.
- **2026-03-15** — Migration uses export/reimport strategy rather than
  `sqlcipher_export` because expo-sqlite doesn't expose the ATTACH...KEY syntax.
  Safe for the expected dataset size (collection management, not big data).
- **2026-03-15** — Ran `expo prebuild --platform android` to generate native
  project. iOS prebuild requires macOS with Xcode — deferred to CI/EAS Build.
- **2026-03-15** — Removed `/android` and `/ios` from `.gitignore` so native
  folders are committed for EAS Build compatibility.

## Known Gaps

- iOS native folder not yet prebuilt (requires macOS). EAS Build handles
  this during cloud builds via the config plugin.
- If SecureStore is wiped (factory reset, backup restore without keychain),
  the encryption key is lost and the database is unrecoverable. This is a
  fundamental property of device-level encryption, not a bug.
- Expo Go no longer works after prebuild. All testing must use development
  builds via `npx expo run:android` or EAS Build.
- The migration reads all data into memory. For extremely large databases
  (>100 MB), this could cause OOM. Not a concern for the expected use case.
